# Schema Comparison: QBO-Derived Schema vs Kuso Hub App

> Systematic comparison of the data schema derived from QBO JSON extracts against the actual Kuso Hub app implementation (database tables, Edge Functions, hooks, and design specifications).

---

## Executive Summary

The app's implementation is **architecturally more sophisticated** than the flat QBO-derived schema in several important ways — it has grade-based cost reallocation (RSVA), a richer stock unit lifecycle, and proper staged integration patterns. However, there are **significant gaps in payout fee granularity, sale-level fee linkage, and per-unit profit calculation** that mean the app currently cannot answer the question "what was the actual net profit on this specific unit/sale/customer?" with full accuracy.

---

## 1. Entity-by-Entity Comparison

### 1.1 Products / SKU Master

| Aspect | QBO Schema | App (Kuso Hub) | Verdict |
|---|---|---|---|
| **Product identity** | Single `product` table keyed by QBO Item Id, SKU = MPN.grade flat | Two-tier: `product` (MPN-level) + `sku` (MPN.grade variant) | **App is better.** Separating MPN from grade allows shared metadata (images, SEO, dimensions) at product level with grade-specific pricing at variant level. |
| **Grade model** | Derived from SKU string, integer 1–5 | Enum on `sku.condition_grade` ('1'–'5') with grade 5 = non-saleable | **Equivalent.** Both parse from same SKU format. |
| **Pricing fields** | `unit_sale_price`, `unit_purchase_cost` only | `sale_price`, `floor_price`, `avg_cost`, `cost_range`, `market_price` | **App is better.** Floor price with margin guardrail, cost range visibility, and market reference pricing are absent from QBO. |
| **Catalogue enrichment** | None — QBO is just name + price | Full LEGO catalogue: dimensions, weight, EAN, BrickEconomy ID, BrickLink item number, release/retired dates, minifig count | **App is better.** QBO is a financial system, not a product catalogue. |
| **QBO linkage** | `qbo_item_id` as primary key | `sku.qbo_item_id` as cross-reference | **Equivalent intent**, but app correctly treats QBO ID as a reference rather than primary key. |

**Issue: None.** The app's product/SKU model is a superset of the QBO-derived schema.

### 1.2 Stock Units

| Aspect | QBO Schema | App (Kuso Hub) | Verdict |
|---|---|---|---|
| **Granularity** | One row per physical unit | One row per physical unit | **Equivalent.** Both are unit-level. |
| **Lifecycle states** | Simple: `in_stock`, `sold`, `returned`, `written_off`, `adjusted` | Rich: `purchased → graded → listed → sold → shipped → delivered → payout_received → complete` + return branch | **App is better.** 10+ states vs 5. Enables operational visibility into fulfilment pipeline. |
| **Landed cost** | `unit_purchase_price + apportioned_ancillary_cost` (one-step) | Two-step: initial apportionment at purchase, then RSVA reallocation at grading | **App is better.** RSVA redistributes costs by market value when units grade differently — gives more accurate per-unit economics. |
| **FIFO mechanism** | Schema-level `fifo_sequence` integer | DB function `v2_consume_fifo_unit()` using `ORDER BY created_at ASC` with `SELECT FOR UPDATE SKIP LOCKED` | **App is better.** Concurrency-safe FIFO with row locking vs a sequence number. |
| **Payout linkage** | `sale_line_id` FK only | `order_id` FK + `payout_id` FK directly on stock_unit | **App is better.** Direct payout linkage on the unit enables end-to-end lifecycle tracking. |
| **Carrying value** | Manual calculation | `GENERATED ALWAYS AS (landed_cost - accumulated_impairment)` | **App is better.** Computed column ensures consistency. |
| **UID** | Not present | `uid` format "PO052-01" auto-generated by trigger | **App is better.** Human-readable unit identification for operational use. |

**Issue: None.** The app's stock unit model is strictly superior.

### 1.3 Purchases / Procurement

| Aspect | QBO Schema | App (Kuso Hub) | Verdict |
|---|---|---|---|
| **Structure** | `purchase_order` → `purchase_line` (flat, mirrors QBO Purchase) | Dual: legacy `inbound_receipt` → `inbound_receipt_line` (QBO landing) AND v2 `purchase_batches` → `purchase_line_items` (operational) | **App has more moving parts.** The legacy/v2 split adds complexity but supports the staged integration pattern. |
| **Vendor model** | Dedicated `vendor` table with `vendor_type` classification | `supplier_name` TEXT on purchase batch — **no vendor entity** | **QBO schema is better here.** The app has no first-class vendor entity. Vendor is just a text string on the batch, so you can't query "all purchases from John Pye" reliably (name variations), track vendor payment terms, or classify vendors by type. |
| **Non-stock line handling** | Explicit `line_type` enum ('item' vs 'expense'), `expense_category` classification | `inbound_receipt_line.is_stock_line` boolean separates stock from non-stock | **Similar intent, QBO schema has richer classification.** The app's boolean is sufficient for apportionment but doesn't classify expense types (delivery vs buying fee vs other). |
| **Cost apportionment** | Pro-rata by unit value, stored on `stock_unit.apportioned_ancillary_cost` | Pro-rata by unit cost via `v2_calculate_apportioned_costs()` DB function + RSVA reallocation at grading via `v2_reallocate_costs_by_grade()` | **App is better.** Two-step process (apportion then reallocate by grade) is more accurate. |
| **Shared costs** | Derived from account-based expense lines | Explicit `shared_costs` JSONB: `{ shipping, broker_fee, other }` | **App is slightly better.** Named cost buckets are clearer than relying on QBO account names. |
| **VAT treatment** | Not modelled | `supplier_vat_registered` boolean determines whether landed cost is ex-VAT | **App is better.** Critical for UK VAT compliance — reclaimable VAT must be excluded from landed cost. |

**Issues:**

1. **CRITICAL: No vendor entity.** The app stores `supplier_name` as free text. Consequences:
   - Cannot reliably aggregate purchases by vendor (name typos, variations)
   - Cannot track vendor-level metrics (total spend, order frequency, average lead time)
   - Cannot link fee-generating entities (eBay, Stripe) to a canonical vendor record for payout reconciliation
   - **Recommendation:** Create a `vendor` table and FK from `purchase_batches.supplier_id`. Backfill from existing `supplier_name` values.

2. **MODERATE: Expense category classification.** The app's `is_stock_line` boolean can't distinguish between delivery charges and buying fees within the non-stock lines. Both get lumped into `shared_costs`. This is fine for apportionment but limits reporting on "how much am I spending on auction fees vs shipping?"
   - The QBO data clearly separates these via account names (`Cost of sales:Courier and delivery charges` vs `Cost of sales:Buying Fees`)
   - **Recommendation:** Add `cost_category` enum to `purchase_line_items` or break `shared_costs` JSONB into more granular buckets.

### 1.4 Sales / Orders

| Aspect | QBO Schema | App (Kuso Hub) | Verdict |
|---|---|---|---|
| **Order model** | `sale` → `sale_line` (mirrors QBO SalesReceipt) | `sales_order` → `sales_order_line` with rich status lifecycle | **App is better.** Operational statuses, fulfilment tracking, carrier/tracking info. |
| **COGS tracking** | Via junction table `sale_line_stock_unit` with `unit_cogs` | `sales_order_line.cogs` column + `stock_unit_id` FK | **Equivalent intent.** App stores COGS directly on the line. QBO schema uses a junction table which supports multi-unit lines more cleanly, but the app's approach works since most sales are qty=1. |
| **Discount handling** | Separate `discount_amount` and `discount_percent` on sale | `discount_total` on order + `line_discount` on line | **Equivalent.** |
| **Channel identification** | Derived from email pattern or deposit account | Explicit `origin_channel` enum + `channel_ids` JSONB | **App is better.** Explicit rather than inferred. |
| **QBO sync** | Not modelled (QBO is source) | Full bi-directional: `qbo_sales_receipt_id`, `qbo_sync_status`, retry logic | **App is better.** Designed for operational sync, not just data extraction. |
| **Customer linkage** | FK to `customer` table | FK to `customer` table + `user_id` for registered website users | **App is better.** Dual identity model (QBO customer + auth user). |

**Issues:**

3. **MODERATE: No junction table for multi-unit sale lines.** The app has `sales_order_line.stock_unit_id` as a single FK. If a sale line has `quantity > 1`, only one stock unit can be linked. The QBO data shows most sales are qty=1, but the schema should support multi-unit lines.
   - In practice, the app's `v2-process-order` function creates separate order lines per unit during FIFO allocation, so this works — but it means the DB schema diverges from what QBO sends (one line with qty=3 becomes three lines with qty=1).
   - **Consequence:** Reconciliation between QBO SalesReceipt line counts and app order line counts will always differ for multi-unit sales.

### 1.5 Returns / Refunds

| Aspect | QBO Schema | App (Kuso Hub) | Verdict |
|---|---|---|---|
| **Return model** | Dedicated `return` + `return_line` tables with `return_disposition` enum | **No dedicated return tables.** Returns handled via order status transitions (`return_pending → refunded / restocked`) and stock unit status changes | **QBO schema is more explicit.** |
| **Refund tracking** | Links to original sale, tracks refund amount and tax | QBO RefundReceipt synced via `qbo-sync-refund-receipt` function, but no local refund entity | **Gap in app.** |
| **Restock logic** | `return_disposition = 'restock'` reverts unit to `in_stock` | Stock unit transitions to `restocked` status | **Equivalent intent.** |
| **QBO linkage** | `qbo_refund_id` on return record | RefundReceipt created in QBO via Edge Function, but no local ID stored back | **Gap.** No way to reconcile app refunds to QBO refunds after the fact. |

**Issues:**

4. **SIGNIFICANT: No local refund/return entity.** The app handles returns entirely through status transitions on existing `sales_order` and `stock_unit` records. Consequences:
   - Cannot track partial refunds (2 of 3 items returned) as a discrete event
   - Cannot record the refund amount separately from the original order total
   - Cannot link a refund to the specific payout it was deducted from
   - No audit trail of when and why refunds were issued
   - **Recommendation:** Create `refund` and `refund_line` tables that link back to `sales_order` and `stock_unit`, storing the refund amount, reason, and QBO RefundReceipt ID.

### 1.6 Payouts / Deposits

| Aspect | QBO Schema | App (Kuso Hub) | Verdict |
|---|---|---|---|
| **Payout model** | `payout` table with `gross_amount`, `total_deductions`, `net_amount` | `payouts` table with `gross_amount`, `total_fees`, `net_amount` | **Equivalent.** |
| **Sale linkage** | `payout_sale_link` junction table | `payout_orders` junction table (payout → sales_order) | **Equivalent.** |
| **Fee model** | `payout_fee` + `payout_fee_line` — each fee purchase decomposed into individual categorised lines with optional sale linkage | `payouts.fee_breakdown` JSONB: `{ fvf, promoted_listings, international, processing }` — aggregate totals only | **QBO schema is significantly more granular.** See Issue #5 below. |
| **Fee-to-sale linkage** | `payout_fee_line.linked_sale_id` enables per-sale fee attribution | **Not implemented.** Fees are aggregate per payout, not linked to individual sales | **Critical gap.** See Issue #5. |
| **Provider identification** | `payout_provider` enum derived from linked vendors | `channel` enum ('ebay', 'stripe') | **Equivalent.** |
| **QBO sync** | Not modelled (QBO is source) | `qbo_deposit_id`, `qbo_expense_id`, `qbo_sync_status` | **App is better for operational sync.** |
| **Reconciliation** | Reconciliation check query comparing expected vs actual | `reconciliation_status` field + `v2-reconcile-payout` function | **App has operational reconciliation; QBO schema has verification query.** Both needed. |

**Issues:**

5. **CRITICAL: Payout fee granularity and sale-level attribution.** This is the single biggest gap between the QBO data and the app's model.

   **What QBO shows:** Each deposit (payout) contains linked Purchase records that itemise every fee and shipping label. The eBay Purchase records break down into individual lines — e.g., 10 shipping labels at £2.45 each + 1 selling fee line. These are traceable to specific sales within the same payout.

   **What the app stores:** A single JSONB blob `fee_breakdown: { fvf: 45.43, promoted_listings: 0, international: 0, processing: 0 }` at the payout level. No per-sale fee allocation. No shipping label tracking. No distinction between sale-level fees and platform-level fees.

   **Consequences:**
   - **Cannot calculate accurate per-sale profit.** Without knowing which fees belong to which sale, profit calculations must use estimated fee rates rather than actual deductions.
   - **Cannot identify which sales had shipping labels purchased via eBay.** This is the single largest variable cost per sale after COGS.
   - **Cannot distinguish sale-level fees (commission, postage) from platform fees (advertising, subscriptions).** Both are lumped together.
   - **Payout reconciliation is weaker.** If a payout has a discrepancy, you can't drill into which specific fee or sale caused it.
   - **QBO reconciliation has no local counterpart.** The QBO Deposit has Purchase-level detail that the app doesn't mirror.

   **Recommendation:**
   - Create `payout_fee` table (one row per fee purchase within a payout)
   - Create `payout_fee_line` table (one row per fee line within a fee purchase)
   - Add `linked_sale_id` FK on fee lines for sale-level fees
   - Add `fee_category` enum: `selling_fee`, `shipping_label`, `payment_processing`, `advertising`, `subscription`, `buying_fee`
   - During eBay payout import, decompose transaction-level fees to individual line items
   - For Stripe, link processing fee to the specific sale (Stripe fee = per-charge)

## 2. Cost Calculation Comparison

### 2.1 Landed Cost Apportionment

| Aspect | QBO Schema | App | Match? |
|---|---|---|---|
| **Method** | Pro-rata by unit value | Pro-rata by unit cost | **Near-equivalent.** Both use proportional allocation. The app uses `unit_cost` while QBO schema uses `unit_price` — effectively the same thing. |
| **Formula** | `share = unit_price / stock_subtotal × ancillary_total` | `share = (unit_cost × qty) / total_unit_costs × total_shared_costs` | **Match** (per-unit). |
| **Grade reallocation** | Not modelled | RSVA: redistributes by market value at grading time | **App is better.** RSVA handles the real-world scenario where units from the same purchase grade differently and should carry costs proportional to their sales potential. |
| **Storage** | `stock_unit.apportioned_ancillary_cost` + `stock_unit.landed_cost` | `purchase_line_items.apportioned_cost` + `purchase_line_items.landed_cost_per_unit` + `stock_unit.landed_cost` (after RSVA) | **App has better traceability.** Cost stored at both line and unit level, with RSVA override at unit level. |

### 2.2 FIFO Costing

| Aspect | QBO Schema | App | Match? |
|---|---|---|---|
| **Selection criteria** | `fifo_sequence` integer, ascending | `created_at ASC` with `SELECT FOR UPDATE SKIP LOCKED` | **App is better.** Concurrency-safe, no manual sequence management. |
| **COGS recording** | Junction table `sale_line_stock_unit.unit_cogs` | `sales_order_line.cogs` column | **Equivalent.** |
| **Trigger** | Sale line insert | `v2-process-order` Edge Function | **Equivalent.** |

### 2.3 Weighted Average Cost

| Aspect | QBO Schema | App | Match? |
|---|---|---|---|
| **Calculation** | Described for reporting only | `v2_recalculate_variant_stats()` updates `sku.avg_cost` on every status change | **App is better.** Actively maintained, triggers on grade/sell/restock events. |

### 2.4 Floor Price

| Aspect | QBO Schema | App | Match? |
|---|---|---|---|
| **Concept** | Not modelled | `floor = highest_batch_cost × (1 + margin_target)` with configurable margin (default 25%) | **App-only feature.** Important operational guardrail absent from QBO model. |

## 3. Per-Unit Profit Calculation

This is the core output the QBO-derived schema was designed to produce. Here's how the app compares:

### QBO Schema: `unit_profit_view`

```
net_profit = net_revenue - landed_cost - selling_fee - shipping_cost - payment_processing_fee
```

All components available because payout fees are linked to specific sales.

### App: No Equivalent View or Function

The app can calculate:
- **Revenue per unit**: `sales_order_line.unit_price` (known)
- **COGS per unit**: `sales_order_line.cogs` = `stock_unit.landed_cost` (known, FIFO)
- **Gross profit per unit**: revenue - COGS (known)

The app **cannot** currently calculate:
- **Selling fee per unit**: Only aggregate `payouts.fee_breakdown.fvf` at payout level — not attributed to individual sales
- **Shipping cost per unit**: Outbound shipping labels are in the payout fee aggregate — not linked to the sale they fulfil
- **Payment processing fee per unit**: Stripe fees are aggregate per payout — not linked to the specific charge

**Consequence:** The app can report **gross margin** (revenue minus FIFO cost) but not **net margin** (after channel fees, shipping, and payment processing). For a resale business where channel fees typically consume 12–15% of revenue and shipping another 5–10%, this means the reported margin overstates actual profitability by 17–25%.

## 4. Data Flow Gaps

### 4.1 QBO → App (Inbound)

| QBO Entity | App Landing Table | Promotion Target | Gap? |
|---|---|---|---|
| Item | `landing_raw_qbo_item` | `product` + `sku` | No |
| Purchase (stock) | `landing_raw_qbo_purchase` | `inbound_receipt` → `purchase_batches` | No |
| Purchase (expense) | `landing_raw_qbo_purchase` | **Not promoted** — expenses with no stock items are landed but ignored | **Minor gap.** Operating expenses from QBO are not tracked in the app. Not needed for stock tracking but means the app can't report total operating costs. |
| SalesReceipt | `landing_raw_qbo_sales_receipt` | `sales_order` + `sales_order_line` | No |
| RefundReceipt | `landing_raw_qbo_refund_receipt` | **Partial** — processed by `qbo-process-pending` but no dedicated local entity | **Gap.** See Issue #4. |
| Deposit | **No landing table for QBO deposits** | `payouts` (created from eBay API, not QBO) | **Significant gap.** See Issue #6 below. |
| Customer | `landing_raw_qbo_customer` | `customer` | No |
| Vendor | **No landing table** | **No vendor entity** | **Gap.** See Issue #1. |

**Issues:**

6. **SIGNIFICANT: QBO Deposits not ingested.** The app creates payout records from the **eBay Finances API** and manual Stripe entries — not from QBO Deposits. But the QBO Deposit is the authoritative record of what actually hit the bank account, including the exact fees deducted.

   **Consequences:**
   - No reconciliation between eBay's reported payout and QBO's recorded deposit
   - Stripe payouts are manual entries — no automated import
   - The 97 QBO deposits contain richer fee detail (linked Purchase records with itemised lines) than what the eBay API provides
   - If eBay's API and QBO's deposit disagree on the amount, there's no detection mechanism

   **Recommendation:** Add `landing_raw_qbo_deposit` table. Ingest QBO deposits and cross-reference with app's `payouts` table. Flag discrepancies. Use QBO deposit detail as authoritative source for fee decomposition.

### 4.2 App → QBO (Outbound)

| App Event | QBO Entity Created | Gap? |
|---|---|---|
| New SKU graded | QBO Item (via `qbo-sync-item`) | No |
| Order created | QBO SalesReceipt (via `qbo-sync-sales-receipt`) | No |
| Order refunded | QBO RefundReceipt (via `qbo-sync-refund-receipt`) | **Minor gap:** RefundReceipt ID not stored back on local record |
| Payout recorded | QBO Deposit + Expense (via `qbo-sync-payout`) | No |
| Customer created | QBO Customer (via `qbo-upsert-customer`) | No |
| Purchase recorded | **Not synced to QBO** | **Gap:** App-originated purchases (v2 batches) are not pushed to QBO as Purchase transactions. QBO only has purchases that originated from QBO. |

## 5. Summary of All Issues

### Critical (blocks accurate financial reporting)

| # | Issue | Impact | Recommendation |
|---|---|---|---|
| 5 | **No sale-level fee attribution** | Cannot calculate net profit per unit/sale/customer. Margin overstated by 17–25%. | Create `payout_fee` + `payout_fee_line` tables with `linked_sale_id`. Decompose eBay transaction fees to sale level. |

### Significant (impairs reconciliation or reporting)

| # | Issue | Impact | Recommendation |

| # | Issue | Impact | Recommendation |

## 6. What the App Does Better Than the QBO Schema

The comparison isn't one-directional. The app has capabilities the QBO-derived schema doesn't model:

1. **RSVA (Relative Sales Value Allocation)** — Grade-based cost reallocation is a genuinely better cost accounting method than flat apportionment.

2. **Concurrency-safe FIFO** — `SELECT FOR UPDATE SKIP LOCKED` prevents race conditions in concurrent order processing. The QBO schema's `fifo_sequence` integer would need application-level locking.

3. **Floor price guardrails** — Prevents listing below cost + margin. Critical for operational pricing discipline.

4. **Rich stock unit lifecycle** — 10+ states vs 5. Enables operational dashboards, fulfilment tracking, and bottleneck identification.

5. **Bi-directional QBO sync with retry** — `qbo_sync_status`, `qbo_retry_count`, `qbo_last_error` on orders. The QBO schema treats QBO as read-only source.

6. **Channel listing fee estimation** — `channel_listing.estimated_fees` and `channel_listing.estimated_net` pre-calculate expected take-home at listing time. Useful for pricing decisions even if actuals differ.

7. **VAT-aware landed cost** — `supplier_vat_registered` flag correctly excludes reclaimable VAT from landed cost. The QBO schema doesn't model this.

8. **CSV sync infrastructure** — Full staged import/export with diff preview, apply, and rollback. Operational flexibility the QBO schema doesn't consider.

9. **Pricing engine** — Configurable margin targets, markdown schedules, automatic price cascading to channel listings.

10. **Blue Bell club commission** — Business-specific commission tracking for partner sales.

## 7. Recommended Implementation Priority

### Phase 1: Sale-Level Fee Attribution (Issue #5)
This is the highest-value change. Without it, no profit reporting is trustworthy.

1. Create `payout_fee` and `payout_fee_line` tables
2. Modify `ebay-import-payouts` to decompose transaction-level fees
3. Link selling fees and shipping labels to specific `sales_order` records
4. For Stripe: calculate fee from known rate (1.4% + 20p) and link to sale
5. Create `unit_profit_view` materialised view

### Phase 2: Vendor Entity (Issue #1)
1. Create `vendor` table with `vendor_type` classification
2. Backfill from unique `purchase_batches.supplier_name` values
3. Add FK from purchase_batches to vendor
4. Add FK from payout_fee to vendor (for fee-charging entities)

### Phase 3: Refund Entity (Issue #4)
1. Create `refund` + `refund_line` tables
2. Link to original order, stock unit, and payout
3. Modify `qbo-sync-refund-receipt` to store QBO ID back
4. Update return workflow to create refund records

### Phase 4: QBO Deposit Ingestion (Issue #6)
1. Add `landing_raw_qbo_deposit` table
2. Create `qbo-sync-deposits` Edge Function (LAND phase)
3. Add reconciliation logic comparing app payouts to QBO deposits
4. Flag discrepancies for manual review
