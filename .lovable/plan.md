\n# Multi-Source Product Enrichment & Channel Value Mapping\n\n## Goal\n\nTurn the existing single-source enrichment (BrickEconomy + Rebrickable + product table) into a true multi-source pipeline where **BrickEconomy, BrickOwl, BrickSet, and BrickLink** each land in their own holding tables. Surface conflicts on the Specifications tab, let staff pick the winning source per attribute per product (or override entirely), and project the chosen value out to each channel honouring that channel's native attribute schema, allowed values, and custom-attribute capability.\n\nThe existing `canonical_attribute` / `channel_attribute_mapping` / `product_attribute` plumbing already supports most of this conceptually; the work is to extend it for multiple sources, add cross-source field equivalence, and add channel value translation.\n\n---\n\n## ⚠ Hard constraint — BrickEconomy value data is untouchable\n\nBrickEconomy is the sole source of truth for **all value / pricing data** and stays that way. The multi-source pipeline must **not** touch:\n\n- `current_value`, `retail_price`, `growth`, `paid_price` on `brickeconomy_collection`\n- The entire `brickeconomy_price_history` table and chart query\n- `brickeconomy_portfolio_snapshot`\n- The `market_price`, `floor_price`, `cost_range`, `avg_cost` recalculation on `sku` (driven from BrickEconomy + landed cost)\n- The `BrickEconomyPriceChart`, the existing `brickeconomy-sync` price/history/snapshot logic, and the daily 100-call quota tracking\n- Any pricing-engine code that reads BrickEconomy values\n\nConcretely:\n\n- Canonical attributes flagged `attribute_group = 'value'` (a new group covering `retail_price`, `current_value`, `growth`, `market_price`, etc.) are **locked to BrickEconomy** — they get no provider chain, no source picker, no per-product override UI, no channel value mapper.
