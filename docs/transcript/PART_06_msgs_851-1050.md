### #983 User
**Major diagnostic prompt** — "Stock Unit Inventory Diagnostic". Asked for full investigation of how `qty_on_hand` is calculated, what happens during order creation per channel (eBay/Stripe/manual), what happens during goods-in, integrity SQL queries, QBO sync gap analysis, root cause hypothesis, and recommended fix approach. ### #984 AI (huge response)
Findings:
- `qty_on_hand` is computed from `v2_variant_stock_summary` view counting `v2_status IN ('graded', 'listed')`.
- Two parallel status systems: legacy `status` and `v2_status`. Order-consumption code (Stripe webhook, eBay process-order Step 9) sets legacy `status = 'closed'` but never updates `v2_status` → sold units stay counted as on-hand.
- Stripe webhook FIFO sets `stock_unit_id` on lines but skips `v2_status`.
- eBay processor runs legacy FIFO (Step 9), then fires `v2-process-order` separately — race condition.
- Cash sales correctly use `v2_consume_fifo_unit` which updates both columns.
- QBO correctly decrements stock via SalesReceipts but app's view ignores this because `v2_status` is unchanged.

Recommended fixes: update Stripe webhook + ebay-process-order to also set `v2_status='sold'`/`sold_at`, repair existing data with SQL, deprecate legacy `status` long-term.
