## 6. Payouts Deep Dive — Reconciliation, Counts, Slide-out → Page (msgs 1299–1316)

### Root causes identified
- Slide-out unit count read from `stock_unit.payout_id`, but `v2-reconcile-payout` only transitioned units in `delivered` status — leading to "9 orders / 0 units".
- `qbo-sync-payout` used hard-coded account refs `"1"`/`"2"` and mismatched fee-breakdown keys; surfaced no real QBO error.
- Stripe `payout.paid` events were unreliable; no backfill path.

### Reconciliation rewrite (`v2-reconcile-payout`)
- Linked `payout_id` and transitioned units in any of `sold | shipped | delivered`.
- Set `payouts.unit_count` from total linked units (regardless of status).
- Tracked `unitsTransitioned` separately for the response payload.

### `qbo-sync-payout` hardening
- Replaced hardcoded refs with `qbo_account_mapping` lookups (selling_fees, undeposited_funds, payout_bank, etc.).
- Normalized fee breakdown keys across eBay import / Stripe / manual paths.
- Persisted real QBO error string on `payouts.qbo_sync_error`.

### Slide-out → list+detail page refactor
- New `src/pages/admin-v2/PayoutDetailPage.tsx` (thin wrapper).
- New `src/components/admin-v2/PayoutDetail.tsx` extracted from `PayoutDetailSheet`.

- `usePayout(payoutId)` hook added; `App.tsx` route `/admin/payouts/:payoutId` added.

### Transactions card
- New `usePayoutTransactions(externalPayoutId)` — fetches `ebay_payout_transactions` joined to `sales_order` via `matched_order_id`, returns `appGross` for variance.
- `PayoutDetail` renders Transactions table with `TransactionTypeBadge` (SALE green / SHIPPING_LABEL blue / NON_SALE_CHARGE amber / TRANSFER zinc), variance highlighting, clickable matched-order links and footer totals.

## 8. QBO Readiness — Source of Truth Change (msgs 1331–1342)

**Problem:** Readiness still reported "9 orders" because the hook queried `payout_orders`. Sale fees were also excluded from the expense count.

**Plan executed:**
- Hook switched to `ebay_payout_transactions` filtered by `transaction_type != 'TRANSFER'`.
- For SALE rows with null `matched_order_id`, fall back to matching `sales_order.origin_reference == tx.order_id` to find existing `qbo_sales_receipt_id`.
- Expenses now count **all** non-TRANSFER transactions needing a Purchase — including SALE rows with `total_fees > 0`.
- Migration: `ALTER TABLE ebay_payout_transactions ADD COLUMN qbo_purchase_id text;`
- New `PayoutQBOReadiness` interface: `{totalOrders, syncedOrders, unsyncedOrders, totalExpenses, createdExpenses, pendingExpenses}`. `ready = syncedOrders === totalOrders`.

**Edge function rewrite (`qbo-sync-payout`):**
- Per-transaction Purchase creation:
  - **SALE** fees → `selling_fees` account, eBay vendor, customer ref.
  - **SHIPPING_LABEL** → `selling_fees`, eBay vendor.
  - **NON_SALE_CHARGE** → `subscription_fees` (fallback `selling_fees`), eBay vendor, memo.
- All with 20% VAT.
- Stores QBO purchase id back on `ebay_payout_transactions.qbo_purchase_id`.

## 9. The Deposit Error 6000 War (msgs 1351–1372, 1377–1396, 1399–1450)

Eight (or more) consecutive failed attempts to push a Deposit to QBO with the same misleading error: *"Select a bank account for this deposit."*

### Iterations attempted
1. **Exclude NON_SALE_CHARGE from deposit lines** — payload total mismatch hypothesis. Did not fix.
2. **Verify bank ref `1150040004 / Current`** — confirmed valid Bank/Checking. Not the cause.
3. **Add `LinkedTxn[].TxnType: "Payment"`** — still 6000.
4. **Pull existing QBO deposits** via new `qbo-sync-deposits` button on `QboSettingsCard` to inspect real payload structure.

### Real QBO Deposit shape (discovered)
```json
{
  "Line": [
    { "Amount": 69.95,
      "DepositLineDetail": { "PaymentMethodRef": { "value": "3" } },
      "LinkedTxn": [{ "TxnId": "631", "TxnLineId": "0", "TxnType": "SalesReceipt" }] },
    { "Amount": -1.95,
      "DepositLineDetail": { "PaymentMethodRef": { "value": "1" } },
      "LinkedTxn": [{ "TxnId": "652", "TxnLineId": "0", "TxnType": "Purchase" }] }
  ]
}
```

### Three structural fixes applied (msgs 1379–1390)
1. `TxnType: "Purchase"` for negative fee lines (not "Payment").
2. `TxnLineId: "0"` on every `LinkedTxn`.
3. **Drop `AccountRef`** from `DepositLineDetail` on linked lines, replace with `PaymentMethodRef: { value: "1" }`. QBO infers the account from the linked txn.
4. **Removed lump-sum fallback** — function now returns 422 if no SalesReceipt lines exist (per user direction: any such case is a genuine fault).

### External ID alignment (msgs 1391–1392)
- Outbound: `DocNumber = externalPayoutId` (e.g. `7388684270`) on QBO Deposit.
- Inbound (`qbo-process-pending`): extract `external_payout_id` from `deposit.DocNumber`, fall back to memo regex `payout\s+(\S+)`.

### UI cleanup + DocNumber on expenses (msgs 1393–1398)
- Removed App Gross / Variance columns from transactions table.
- Status pill now flows: Unmatched → Matched → Synced (using `qbo_purchase_id`).
- SALE expense Purchases now carry the order number as `DocNumber`.
- `CustomerRef` added to each `AccountBasedExpenseLineDetail` for SALE expense lines (resolves `customer_id` → `qbo_customer_id` via batched lookup).

### The £222.06 vs £221.19 / £188.67 discrepancy (msgs 1399–1442)
**Math walkthrough for payout `7388684270`:**
- Sales gross: £301.89, SALE fees: -£46.55, SHIPPING: -£33.16, Insertion fee NON_SALE_CHARGE: -£0.12, Subscription NON_SALE_CHARGE: -£32.40, TRANSFER_FROM: +£32.40 → **£222.06**.

**User clarifications shaping the design:**
- Insertion fees carry `references[].referenceType = 'item_id'` (e.g. `205984594775`) — should be allocated as COGS to the specific listing, not bundled into subscription account.
- Subscription `NON_SALE_CHARGE` is paid from undeposited funds; its TRANSFER mirror **is not a fund movement** but the payment side of the subscription expense.
- Both legs must be created in QBO regardless of net-zero impact, for VAT reclamation and audit.

**Reset Sync UI added** (msgs 1411–1428):
- New section on `PayoutDetail` with three destructive-outline buttons: **Reset Expenses**, **Reset Deposit**, **Reset All**.
- New `useResetPayoutSync` mutation + `admin-data` action `reset_payout_sync` with `scope: 'expenses' | 'deposit' | 'all'` clearing `qbo_purchase_id`, `qbo_deposit_id`, and resetting `qbo_sync_status` to `pending`.

**Final state at end of Part 8:** Still iterating — discovered that QBO silently drops unlinked deposit lines when linked txns are present (so the +£32.40 TRANSFER vanishes), and that order `24-14326-19004` has a £1.00 mismatch between eBay gross (£15.99) and app `gross_total` (£14.99) reflected in the SalesReceipt. Plan to switch deposit line `gross` source from `tx.gross_amount` to `so.gross_total`, and create the TRANSFER as a standalone subscription Purchase paid from undeposited funds. **Not yet fully resolved — continues into Part 9.**
