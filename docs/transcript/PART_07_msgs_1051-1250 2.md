## 1. Duplicate Purchases, Stale Customers & Missing Payouts (#1051–1056)

A long arc of compounding QBO sync defects. Three root causes were identified and fixed in sequence:

- **Duplicate `purchase_batches`**: Rebuild was deleting `inbound_receipt`/`stock_unit`/`sku` but never `purchase_batches` or `purchase_line_items`. Result: 173 batch rows for 87 distinct QBO refs (every purchase duplicated).
- **Stale customers**: `qbo-sync-customers` queried `SELECT * FROM Customer` with no `Active = true` filter, and `processCustomers` had no orphan-cleanup step.
- **Missing payouts/deposits**: No `qbo-sync-deposits` function existed at all; eBay payout landing tables were emptied on rebuild and never repopulated.

**Fixes implemented:**
- `admin-data` rebuild Phase 2 now deletes `purchase_line_items` then `purchase_batches`, and clears `landing_raw_qbo_deposit`.
- `qbo-sync-customers` filters `Active = true`; `processCustomers` cleans up canonical customers whose `qbo_customer_id` no longer appears in landing.
- New `landing_raw_qbo_deposit` table + `qbo-sync-deposits` edge function + `processDeposits` handler that links deposit lines to `sales_order` via `qbo_sales_receipt_id`.
- `QboSettingsCard` rebuild gained a Phase 2g landing step.

User then escalated — purchases were cleared but never reprocessed. The AI proposed a strict rebuild sequence: **Vendors → Customers → Items → Purchases → Sales/Refunds → Deposits**, with entity-specific draining (`drainPending(entityType)`) so each phase fully completes before advancing. Safety check: stop before Sales if Purchase records have errors.

A second core defect: `processPurchases()` only created `inbound_receipt` + `stock_unit` rows but never the v2 model (`purchase_batches`, `purchase_line_items`) used by the Purchases UI. The only path that previously created batches was the client-side `use-intake.ts` mutation, which rebuild never invoked.

**Unification:**
- Moved purchase promotion into the `process-receipt` edge function (single canonical server-side routine, idempotent per QBO purchase).
- `qbo-process-pending` now calls this routine after creating the inbound receipt; on failure, it rolls back batch + line items + units.
- `use-intake.ts` was rewritten to invoke `process-receipt` instead of writing batches/units directly from the browser.

## 3. Sales Reconciliation: Net vs Gross (#1073–1082)

User flagged that QBO sales receipts showed different totals in the app vs QBO (e.g. QBO 631 = £69.95 in report vs £81.61 in app, diff = £11.66 = the recorded `TotalTax`).

**Root cause (1st bug):** `reconcile-sales` compared QBO `TotalAmt` (which is **net** because all receipts use `GlobalTaxCalculation: TaxExcluded`) against the app's VAT-inclusive `gross_total`. They could never match.

**1st fix:** `qboTotal = TotalAmt + TxnTaxDetail.TotalTax`.

User then reported regression: "Purchases: 537 missing in app, Sales: 220 mismatched."

**Two follow-up bugs found:**
1. Reconcile-purchases was querying `SELECT * FROM Purchase` (all 623 records including expenses, checks, credit cards) but the processor intentionally only imports purchases with `ItemBasedExpenseLineDetail` lines (inventory only).
2. The new sales fix double-counted tax for `TaxInclusive` receipts where `TotalAmt` already includes tax.

**2nd fix:**
- Reconcile-purchases now filters `qboRecords` to those with at least one `ItemBasedExpenseLineDetail` line.
- Reconcile-sales checks `GlobalTaxCalculation` per receipt: `TaxInclusive` → use `TotalAmt` as-is; `TaxExcluded` → add `TotalTax`.

## 4. Ghost Stock Units & Stuck Purchases (#1083–1116)

Multiple cycles trying to clean up SKU `10349-1.1` and similar — units shown but with no item numbers, no costs, no proper allocations.

**Root causes uncovered iteratively:**

1. **`reconcile-stock` was the source of ghost units.** Lines 1548–1567 auto-backfilled stock when QBO qty exceeded app qty, inserting rows with `landed_cost = 0`, `batch_id = NULL`, `line_item_id = NULL`, `v2_status = NULL`. Every reconciliation regenerated the ghosts that had just been cleaned up.
2. **Cleanup logic targeted wrong FK.** `cleanup-ghost-units` did `eq("inbound_receipt_line_id", receipt.id)` but `stock_unit.inbound_receipt_line_id` references `inbound_receipt_line.id`, not `inbound_receipt.id`.
3. **Purchases 881 and 1733 stuck in `error`** with `stock_unit_uid_key` duplicate-key violations.
4. **UID truncation bug in PostgreSQL.** Both `v2_generate_stock_unit_uid()` and `v2_reserve_stock_unit_uids()` used `lpad(..., 2, '0')`, so unit `100` truncated to `10` → `PO583-100` collided with `PO583-10`.
5. **Trigger/manual counter ownership conflict.** The DB trigger incremented `unit_counter` per row while processors also manually overwrote it.

**Fixes (cumulative):**
- `reconcile-stock` made report-only — never inserts ghost units; only flags discrepancies for admin review.
- New `resetQboPurchase` helper in `admin-data/index.ts` correctly maps `inbound_receipt_line.id` values → deletes non-sold stock units, then `purchase_line_items`, then `purchase_batches` (matched by `reference = qboPurchaseId`), then resets landing to `pending`.
- New `reset-qbo-purchase` admin action + "↺ Reset & Retry" amber button in `StagingErrorsPanel.tsx` for QBO duplicate-key errors.
- New PostgreSQL function `v2_reserve_stock_unit_uids(p_batch_id, p_count)` atomically increments `unit_counter` and returns the reserved UID array. Both `qbo-process-pending` and `process-receipt` now pre-allocate UIDs and pass them on insert. Manual `unit_counter` overrides removed.
- Removed `lpad` truncation: `'PO' || batch_num || '-' || seq_num::text`.
- `processPurchases()` now also clears any existing `purchase_batches` with the same `reference` (cascading delete of non-sold units → line items → batch) **before** inserting, fixing reprocessing duplicates (PO-603 / PO-665 / PO-666 all referencing 1733).

## 5. Sales Orders Missing Line Items (#1115–1118)

QBO sales receipt 1796 (eBay order `14-14455-15040`) committed with zero line items and no allocated stock. **23 orders** had this systematic problem.

**Root cause:** `processSalesReceipts` "match-first" logic (lines 874–913) found existing shell orders by `origin_reference`/`doc_number`/`order_number`, enriched them with QBO fields, and committed — but never created line items or called `allocate_stock_units`. Shell orders were typically created earlier by eBay imports.

**Fix:** All three match branches now check `SELECT COUNT(*) FROM sales_order_line WHERE sales_order_id = matched.id`. If zero, the code sets `matchedOrderId` and falls through to the line-item creation + allocation loop instead of `continue`. The order INSERT block was wrapped to skip when `matchedOrderId` is set. The 23 affected landing records were reset to `pending`.

## 7. QBO Webhook Reliable Auto-Processing (#1133–1138)

User asked why receipts 1797–1801 only processed after manual "Process Pending" while 1794 processed automatically.

**Root cause:** Webhook called `qbo-process-pending` once at end of background task. When QBO sent multiple webhooks in quick succession, webhook A's processor invocation completed before webhook B's data finished landing.

**Fix in `qbo-webhook/index.ts`:** Replaced the single fire-and-forget call with a poll-and-drain loop:
- 3-second initial delay after landing (lets concurrent webhooks finish landing)
- Up to 3 processor calls with `batch_size: 50`
- Loop exits early when `result.has_more === false` or `total_remaining === 0`
- 2-second wait between attempts

## 13. Evri-First Shipping Strategy (#1197–1210)

User uploaded Evri (3-tier frequent seller) and eBay carrier shipping rate cards. Refined requirement: **Use Evri direct rates even for eBay orders** to accumulate volume for higher tiers, unless eBay carrier is substantially cheaper or order value warrants Royal Mail's better insurance. Add Evri international rates.

**Schema changes:**
- `shipping_rate_table` gained `tier text` (nullable: `tier_1`/`tier_2`/`tier_3` for Evri, NULL for non-tiered) and `destination` (`domestic`/`international`).
- `selling_cost_defaults` gained `evri_active_tier`, `evri_tier_{1,2,3}_threshold`, `shipping_prefer_evri_threshold` (default £1.00), `high_value_order_threshold` (default £100).

**Rate population:**
- Full Evri domestic 3-tier card (Postable / Small ≤1kg / Small ≤2kg / Medium ≤5kg / Medium ≤10kg / Large ≤15kg / XL ≤30kg).
- Evri international rates per tier.
- eBay-specific carrier rates (Evri ParcelShop, InPost, DPD, UPS, Yodel, DHL, Royal Mail Intl) on `channel = 'ebay'`.

**Selection logic in `admin-data` and `auto-markdown-prices`:**
1. Look up Evri direct rate filtered by `evri_active_tier`.
2. If channel is `ebay`, also look up eBay carrier rate for the band.
3. Use Evri direct unless eBay rate is cheaper by > `shipping_prefer_evri_threshold`, OR sale price > `high_value_order_threshold` AND a Royal Mail tracked/insured option exists.
4. International orders use Evri international (Royal Mail considered for high-value).

Tier auto-calculation noted as future work — count last calendar month's Evri-only (non-eBay) shipments and update `evri_active_tier`.
