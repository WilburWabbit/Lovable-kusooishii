## 1. The QBO Deposit Penny Saga (msgs 1451–1556)

A multi-day battle to make a single eBay payout (`060ee447-…`) land at exactly **£222.06** in QuickBooks Online instead of £222.07.

### 1.1 Error 6000 — "Transaction cannot be applied to Deposit"
- **Symptom:** `qbo-sync-payout` returned 422 with QBO Validation Error 6000.
- **First theories (wrong):**
  - SalesReceipts missing `DepositToAccountRef` → defaulting to bank instead of Undeposited Funds.
  - Stray `TRANSFER` line (+£32.40) in deposit pointing to a COGS account.
- These plans were declined/abandoned after the user pushed back.

### 1.2 First penny drift (£222.07 vs £222.06)
- Floating-point summing of fees: `13.20 + 9.04 + 4.80 + 8.58 + 9.93` evaluated to `45.55000000000004`.
- **Fix attempt:** Applied `round2()` to each `depositLine.Amount`, tightened tolerance from `0.02` → `0.001`. Did not fully resolve.

### 1.3 Misdiagnosis — `GlobalTaxCalculation: "TaxExcluded"` on Deposit
- Hypothesised QBO was applying a tax adjustment to deposit lines.
- Removed the field from `depositPayload`. Still produced £222.07.

### 1.4 Real root cause #1 — Per-line VAT rounding on Purchases
- `createQBOPurchase()` converted gross fees to ex-VAT line by line, rounding each. £9.93 fee landed as £9.92 Purchase in QBO.
- **Plan approved:** integer-pence math, per-document exact-balancing safeguards.

### 1.5 Real root cause #2 — QBO ignores `TxnTaxDetail.TotalTax` for Purchases
- Confirmed from logs: payload sent `Amount: 7.63 + 0.24 + 0.41 = 8.28` net + `TotalTax: 1.65` = expected £9.93. QBO recomputed tax per-line: `1.53 + 0.05 + 0.08 = 1.66` → £9.94.
- **Built shared helper `supabase/functions/_shared/qbo-tax.ts`** with `buildQBOStableLines` algorithm:
  - Pre-solves so QBO's own `round(Amount × 0.20)` per line sums to target tax.
  - Falls back to a ±£0.01 zero-tax "Rounding adjustment" line using QBO TaxCode `10` (No VAT).
  - Drops `TxnTaxDetail.TotalTax` entirely.
- Verified against 250,000+ test cases.

### 1.6 Real root cause #3 — QBO actually rounds at document level
- Next failure: expected £9.93, QBO returned £9.92 (1p **under**).
- Realised QBO's true rule for Purchases with `TaxExcluded` is `tax = round(sum(taxable_nets) × 0.20)` — single doc-level recompute, **not** per-line.
- Rewrote `qbo-tax.ts`: distribute nets so `sum(line_nets) = round(G/1.2)`, append zero-tax balancer for the ~1/6 cases that still drift.

### 1.7 User frustration: "FFS. NOW WHAT?!"
- Then surfaced a **stale SalesReceipt** (1668 at £14.99 vs canonical £15.99 for `KO-0009323`).
- The eBay payout transaction reported £15.99 but `sales_order.gross_total = £14.99`.
- **Plan approved:** Self-healing — detect stale linked QBO docs, delete + recreate from canonical app values, then build deposit.

### 1.8 Final fix — Per-transaction repair loop
- Refactored `qbo-sync-payout` so for each SALE/expense:
  1. Compare QBO `TotalAmt` vs canonical `ebay_payout_transactions.gross_amount`.
  2. If drift: repair `sales_order.gross_total` + `unit_price` (4dp), delete stale QBO doc, invoke `qbo-sync-sales-receipt`.
  3. Re-verify. Only hard-fail after 3 attempts.
- **Subsequent fixes:**
  - `_shared/qbo-helpers.ts` `authenticateRequest` accepts service role key for function-to-function calls (was failing with "Unauthorized — invalid token").
  - `qbo-sync-sales-receipt` made idempotent on QBO "Duplicate Document Number" — looks up existing by DocNumber and adopts.
  - Repair logic preserved 4dp ex-VAT precision (£15.99 was being rounded to £16.00).

## 4. eBay Webhook Investigation (msgs 1571–1578)

- **Order `22-14511-75136`** picked up by polling job (`ebay-sync`) instead of webhook.
- **First (wrong) diagnosis:** Webhook destination disabled by eBay.
- **Correct diagnosis:** Single missed `ORDER_CONFIRMATION` notification — eBay-side dropout. `ITEM_MARKED_SHIPPED` for the same order arrived 2h later normally.
- **Approved fix:** Safety net in `ebay-notifications/index.ts` — when a shipment topic arrives for an unknown order, synchronously invoke `ebay-process-order` with `source: 'shipping-notification-recovery'`, then re-check before applying tracking. Returns `503` if recovery fails so eBay retries.
- Tags `ebay_notification.payload._recovery = { triggered, succeeded, orderId, error }` for observability.

## 6. eBay Inventory Desync Fix (msgs 1611–1630)

### 6.1 Problem
- 36 of 113 live eBay listings had stale `availableQuantity`. MPNs `31058-1` and `60438-1` showed qty 1 on eBay despite being sold out locally.
- **Root cause:** Only `ebay-process-order` (eBay-originated sales) pushed inventory updates back to eBay. `stripe-webhook`, `v2-process-order`, and admin-data manual flows updated local stock but never told eBay.

### 6.2 Implementation
- **New shared helper `_shared/ebay-inventory-sync.ts`** — `pushEbayQuantityForSkus()`:
  - Counts units by `v2_status IN ('graded', 'listed')` (the legacy `status` column was unreliable — 4 units had `status='available'` but `v2_status='sold'`).
  - `PUT /inventory_item/{sku}` to update quantity.
  - If qty hits 0, calls `POST /offer/{id}/withdraw` to actually end the listing.
- **Wired into:** `stripe-webhook`, `v2-process-order`. Created standalone `sync-ebay-quantity` edge function called from client dialogs (`WriteOffDialog`, `ProcessReturnDialog`).
- **Refactored `ebay-sync` `push_stock`** to use the shared helper.

### 6.3 Backfill + verification
- Ran `push_stock` action — reduced live mismatched listings from 36 → 1.
- `31058-1.1` and `60438-1.1` confirmed `ended` on eBay.

### 6.4 Hardening (msg 1618)
- User reported listings still showing as active. Helper was treating eBay withdraw errors (`25710 / 404`) as success and marking `v2_status='ended'` locally without verifying.
- Added mandatory verification: `GET /offer/{id}` after withdraw to confirm `status != PUBLISHED` before marking ended.
- `ebay-sync` `push_stock` now returns `mismatches_remaining` + sample SKUs.
- `EbaySettingsCard` shows red error toast with SKU details on failure.
- Migration reset ~36 listings from `ended` → `live` to force re-evaluation.

### 6.5 Nightly drift cron (msgs 1615–1616)

## Recurring Themes / Lessons

1. **QBO API quirks dominate:** `TxnTaxDetail.TotalTax` is silently ignored; tax is always recomputed at document level for `TaxExcluded`; deposit line amounts cannot override linked-doc totals.
2. **Canonical truth must be explicit:** the historical order's `gross_total` is canonical for the sale; the eBay payout transaction's `gross_amount` is canonical for the cash settlement; they are allowed to differ.
3. **Self-healing > hard fail:** detect drift → delete stale QBO doc → repair canonical → recreate → re-verify; only abort after 3 attempts.
4. **Cross-channel via reference tables, not amount/timing matching.**
5. **When 5+ incremental fixes target the same subsystem, the architecture needs restructuring** — proven by the modular `PayoutAdapter` refactor finally landing after months of eBay-specific patches.
