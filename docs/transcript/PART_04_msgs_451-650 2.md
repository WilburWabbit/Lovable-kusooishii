## QBO Item Webhook Failures (msgs 451–456)

**[#451 / #453] User:** "QBO purchase receipt ID 1748 contains only new items/SKUs. These came through the QBO webhook, but the new items, created prior to the purchase of the items, did not. Why?"

**[#452 / #454] AI diagnosis:** Item webhooks (IDs 387, 388, 389, 413–416) **were** received but every upsert failed with `there is no unique or exclusion constraint matching the ON CONFLICT specification`. Root cause: `sku_qbo_item_id_unique` was defined as a **partial** unique index (`WHERE qbo_item_id IS NOT NULL`). PostgREST's `.upsert({ onConflict: "qbo_item_id" })` requires a **full** unique constraint — partial indexes are not eligible. Both `qbo-webhook` and `qbo-sync-items` already used the correct `onConflict` clause; only the constraint shape was wrong.

**[#455 User: Approved → #456 AI]** Migration applied:
```sql
DROP INDEX IF EXISTS sku_qbo_item_id_unique;
ALTER TABLE public.sku ADD CONSTRAINT sku_qbo_item_id_unique UNIQUE (qbo_item_id);
```
NULL qbo_item_id stays distinct under PostgreSQL's unique semantics, so multiple unlinked SKUs remain valid. User instructed to click **Sync Items** to re-process 387/388/389.

## QBO SKU is Canonical Truth (msgs 521–530)

**[#521] User:** *(canonical SKU rule, verbatim)*
> The SKU sourced from QBO is the canonical source of truth and should be used for all SKU level mapping.
> SKUs are structured in QBO as `<MPN>.<Condition Grade>`, or just `<MPN>` where the condition grade is irrelevant and can be assumed to be 1 for all other grading related functions.
> The app should split the SKU at the `.` and use the first part as the product's MPN for MPN level matching. The app should use the second part, or the absence of a second part, to determine the condition grade — which is either the numeral after the `.`, or 1 if NULL or missing.
> That means the SKU for items on the website storefront should be the same as the SKU in QBO.
> That means the SKU for items in eBay should be the same as the SKU in QBO.
> There is no need to change the SKU in anyway in this app. All normalisation of a SKU must be removed and the SKU from QBO is used verbatim.

**[#522 AI proposed plan]** Use QBO SKU verbatim everywhere. Migration to back-fill: `UPDATE sku SET sku_code = regexp_replace(sku_code, '-G(\d)$', '.\1')`. Across 7 functions (`qbo-sync-items`, `qbo-webhook`, `qbo-sync-sales`, `qbo-sync-purchases`, `process-receipt`, `ebay-sync`, `ebay-process-order`) replace every `${mpn}-G${grade}` reconstruction with raw QBO `Sku` field. `parseSku()` retained only to derive MPN + condition_grade for internal joins.
