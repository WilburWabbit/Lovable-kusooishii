## Codebase Orientation

Before writing any code, read the repo structure and understand what exists. The app is a Lovable project — React frontend hosted on Lovable, Supabase backend (PostgreSQL + Edge Functions + Auth + Storage).

### Tech Stack (do not deviate)

- **Frontend:** React 18 + Vite + TypeScript + Tailwind CSS 3 + shadcn/ui
- **Backend:** Supabase (PostgreSQL + Edge Functions in Deno + Auth + Storage)
- **State:** TanStack React Query v5 (server state), Zustand (client state — cart/wishlist only)
- **Routing:** react-router-dom v6
- **Payments:** Stripe Checkout Sessions
- **Charts:** Recharts
- **Monospace font:** JetBrains Mono (for data values in admin UI)
- **Body font:** Inter

### Key Directories

### Parallel Build Strategy

This is a parallel build. The new admin (v2) is built alongside the existing admin in the same project. Both coexist and share the same database.

- **Existing admin** lives at `/admin/*` — DO NOT MODIFY any existing admin files
- **New admin** lives at `/admin/v2/*` during development
- **Database:** New tables are additive. Existing tables that both versions use (orders, customers) get new columns added, not replaced. Both versions read from the same database.
- **Components:** All new components go in `src/components/admin-v2/`, pages in `src/pages/admin-v2/`
- **Hooks:** New hooks go in `src/hooks/admin/` — these are new files, not modifications to existing hooks
- **Edge Functions:** New functions get their own directories alongside existing ones
- **Cutover:** When v2 is complete and approved, routes are swapped in `App.tsx` — `/admin/v2/*` becomes `/admin/*`, old admin optionally moves to `/admin/legacy/*`. This is a single commit.

### Conventions to Follow

- All new admin routes go under `/admin/v2/*` during development (swapped to `/admin/*` at cutover)
- Route definitions are in `App.tsx`
- Database types are auto-generated by Supabase — after schema changes, types need regenerating
- Edge Functions handle all external API calls (eBay, QBO, BrickEconomy, etc.)
- React Query hooks wrap all data fetching — no direct Supabase calls in components
- British English in all copy, comments, and UI labels
- Date format: `1 March 2025` (not US format)

### Deployment Model

- Frontend: Lovable auto-deploys from the main branch
- Edge Functions: Lovable does NOT auto-deploy edge functions pushed via Git. There is a GitHub Actions pipeline for this (or use Supabase CLI manually). Supabase project ref: `gcgrwujfyurgetvqlmbf`
- SQL migrations: Run via Supabase CLI or dashboard, not auto-applied from Git

## Implementation Phases

Build in this order. Each phase should result in a working state — no half-built features. Commit after each phase.

### Phase 1: Database Schema

Create the new tables via SQL migration. The spec defines 12 entities (Section 2). Build them in dependency order.

**Migration 1: Core inventory tables**

```sql
-- purchase_batches
-- purchase_line_items
-- stock_units
-- products (if not already existing — check current schema first)
-- product_variants
-- product_images
```

**Migration 2: Commerce tables**

```sql
-- channel_listings
-- orders (may already exist — check and alter if needed)
-- order_line_items
-- customers (may already exist — check and alter if needed)
-- payouts
```

**Critical notes:**

- Check what tables already exist before creating. The current app likely has inventory, orders, and customer tables. Alter and migrate data rather than dropping and recreating.
- All `status` fields should be PostgreSQL enums, not free text.
- Add appropriate indexes: `stock_units(batch_id)`, `stock_units(sku)`, `stock_units(status)`, `stock_units(order_id)`, `order_line_items(order_id)`, `order_line_items(stock_unit_id)`.
- Add RLS policies: all admin tables should require authenticated user with admin role.
- The `stock_units.uid` field (human-readable ID like `PO052-01`) should be auto-generated from the batch ID and a sequence within the batch.
- `product_variants.qty_on_hand` should be a computed column or a database function, not manually maintained. Use a view or function that counts stock units with status in (`graded`, `listed`).
- `product_variants.avg_cost` should be recalculated via a database function triggered on stock unit inserts/updates.

**Migration 3: Functions and triggers**

```sql
-- Function: calculate_apportioned_costs(batch_id) — runs on batch save
-- Function: reallocate_costs_by_grade(line_item_id) — runs when units are graded
-- Function: recalculate_avg_cost(sku) — runs on stock unit insert/update
-- Function: calculate_floor_price(sku) — runs on stock unit insert/update
-- Function: compute_vat(gross_amount) — returns { net, vat } with penny-rounding
-- Trigger: on stock_unit grade change → recalculate landed costs, avg cost, floor price
-- Trigger: on stock_unit status change to 'sold' → validate FIFO consumption
```

## Critical Gotchas

1. **VAT on QBO SalesReceipts:** Always send ex-VAT unit price + explicit VAT amount. Never trust QBO's TaxInclusive setting. Always validate that line totals sum to the gross order total before pushing.

2. **eBay orders have no VAT breakdown.** Calculate: `net = total / 1.2`, `vat = total - net`.

3. **Multi-line order VAT rounding:** Calculate per-line, sum, compare to total, adjust last line by the discrepancy (max 1p).

4. **Floor price must account for highest-cost batch on hand**, not the average. Query: `SELECT MAX(landed_cost) FROM stock_units WHERE sku = $1 AND status IN ('graded', 'listed')`.

5. **FIFO consumption on sale:** Query: `SELECT id FROM stock_units WHERE sku = $1 AND status = 'listed' ORDER BY created_at ASC LIMIT 1`. Update that unit's status to 'sold' and record its `landed_cost` as COGS on the order line item.

6. **Relative sales value allocation** runs when units are graded, not when they're purchased. The initial landed cost from apportionment is a temporary value that gets recalculated.

7. **Edge functions are not auto-deployed by Lovable.** After pushing edge function changes via Git, deploy via CLI or GitHub Actions.

8. **Don't break the public storefront OR the existing admin.** The new admin is at `/admin/v2/*`. The public shop and existing `/admin/*` pages must continue to work throughout. Do not modify any files in `src/components/admin/` or `src/pages/admin/`.

9. **SKU format is `{MPN}.{grade}`** — e.g., `75367-1.1`. The grade is an integer 1–4. There is no sequence number in the SKU.

10. **Products and variants are created lazily.** A product record for an MPN is created when that MPN first enters a purchase. A variant record for a specific grade is created when the first unit of that MPN is graded at that level.
